# תיעוד: שינוי מקור הנתונים של חלון החלוקה — והחזרה

## המצב **לפני** השינוי (מקורי)

- **חלון "נתוני חלוקה"** קרא נתונים מטבלת **`store_products`**.
- הטבלה `store_products` מתאכלסת **רק** כאשר מעלים קובץ **"נתוני מכירות"** (טאב "נתוני מכירות" בהעלאה).
- כל שורה ב-`store_products`: חנות + מוצר + `monthly_qty`, `monthly_sales`, `monthly_returns` (JSONB לפי חודש).
- חלון החלוקה הציג: חנות, מוצר, כמות, חזרות, מכירות, נהג, עיר, רשת וכו'.

## מה **עשיתי** בשינוי (שגוי)

1. **שיניתי את מקור הנתונים** מ-`store_products` ל-**`store_product_weekly`**.
2. `store_product_weekly` מתאכלסת **רק** כאשר מעלים קובץ **"פירוט מוצרים"** (טאב "פירוט מוצרים" בהעלאה).
3. הוספתי:
   - `WeeklyRecord` + `buildRowsFromWeekly()` — קיבוץ שבועי לפי חנות+מוצר+חודש.
   - `maxMonthFromWeekly` / `latestMonthRangeFromWeekly` — לחישוב טווח תאריכים.
4. החלפתי את ה-fetch: במקום `store_products` שאילתה ל-`store_product_weekly`.

## התוצאה (למה "הרסתי")

- אם **הנתונים האמיתיים** שלך מגיעים מ-**"נתוני מכירות"** (ונשמרים ב-`store_products`), אז אחרי השינוי חלון החלוקה **לא קרא יותר** מהטבלה הזו.
- הוא קרא רק מ-`store_product_weekly`, שאצלך כנראה ריקה או לא בשימוש — ולכן **"אין נתונים"**.

## החזרה (מה עשיתי עכשיו)

- **מחזירים** את `useDistributionV2Data` למצב המקורי:
  - קריאה מ-**`store_products`** (כמו לפני).
  - שימוש ב-`buildRows(stores, storeProducts)` עם `monthly_qty`, `monthly_sales`, `monthly_returns`.
  - זריעת פילטר תאריכים מ-`latestMonthRangeFromQty(storeProducts)`.
  - `dataLastDate` מחושב מ-`store_products` (updated_at / חודש אחרון).

**חלון החלוקה יחזור להציג את הנתונים כמו קודם — מהעלאת "נתוני מכירות".**

---

הערה: אם בעתיד תרצה שגם העלאת "פירוט מוצרים" תזין את חלון החלוקה — אפשר להוסיף לוגיקה שמשלבת/מציגה משתי הטבלאות, בלי להחליף לגמרי את `store_products`.
