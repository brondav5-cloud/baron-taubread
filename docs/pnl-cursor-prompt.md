# פיצ'ר רווח והפסד — שכתוב מלא

## הוראה ראשונה — מחיקת הפיצ'ר הישן

**לפני שמתחילים לבנות משהו חדש:**

1. מחק את כל הקומפוננטות, הקבצים, והלוגיקה שקשורים לפיצ'ר רווח והפסד הקיים
2. מחק את טבלאות Supabase הישנות שקשורות לפיצ'ר (אם קיימות):
   - `revenue_entries` (הכנסות ידניות — כבר לא צריך)
   - כל view או function ב-Supabase שקשור לרווח והפסד
3. **אל תמחק** טבלאות שמשמשות פיצ'רים אחרים (בדוק לפני מחיקה)
4. מחק imports ו-routes שמפנים לקומפוננטות שנמחקו
5. ודא שהאפליקציה עדיין רצה בלי שגיאות אחרי המחיקה

## הפילוסופיה

- **הקובץ הוא מקור האמת** — המשתמש מעלה קובץ כרטסת הנהלת חשבונות (Excel) ומתוכו נבנה כל הדוח
- **אין הזנה ידנית של הכנסות או הוצאות** — הכל בא מהקובץ
- **יש שכבת עריכה (override)** — המשתמש יכול לתקן, להוסיף הערה, או לדרוס ערך ספציפי מבלי לשנות את הנתונים המקוריים
- **יש שכבת סיווג מותאם** — 3 שכבות: (1) מפתח סיווג פנימי — override ידני לחשבון ספציפי (גובר על הכל), (2) קיבוצים מותאמים (custom_groups) שמקבצים group_codes, (3) group_code מהקובץ המקורי. בנוסף: תגיות חופשיות ומרכזי עלות
- **הכל ויזואלי** — גרפים, צבעים, אחוזים, זיהוי סטיות. לא רק טבלאות
- **סיווג אחרון = הנכון** — אם חשבון עבר סיווג (group_code) בין קבצים/שנים, הסיווג האחרון הוא ברירת המחדל. כל הנתונים ההיסטוריים מוצגים לפי הסיווג האחרון. יש אפשרות לעבור לתצוגת "מקור" שמראה כל שנה לפי הסיווג המקורי שלה

הקובץ הוא **כרטסת הנהלת חשבונות** בפורמט Excel (.xlsx). מבנה:

עמודה 4: מספר תנועה
עמודה 5: קוד מנה
עמודה 7: חשבון נגדי (counter_account)

## טבלאות Supabase

### uploaded_files
```sql
CREATE TABLE uploaded_files (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  filename TEXT NOT NULL,
  year INTEGER NOT NULL,
  month INTEGER, -- NULL = קובץ שנתי, 1-12 = חודשי
  file_type TEXT NOT NULL CHECK (file_type IN ('yearly', 'monthly')),
  uploaded_at TIMESTAMPTZ DEFAULT now(),
  row_count INTEGER, -- כמה תנועות יובאו
  status TEXT DEFAULT 'processing' CHECK (status IN ('processing', 'completed', 'error'))
);
```

### accounts
```sql
CREATE TABLE accounts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  code TEXT NOT NULL, -- "1000", "2536", "2703"
  name TEXT NOT NULL, -- שם החשבון — מתעדכן אוטומטית לפי הקובץ האחרון שהועלה
  latest_group_code TEXT, -- מתעדכן אוטומטית לפי הקובץ האחרון שהועלה
  account_type TEXT NOT NULL CHECK (account_type IN ('revenue', 'expense')),
  UNIQUE(user_id, code)
);
-- account_type: group_code שמתחיל ב-6 = revenue, כל השאר = expense
-- ⚠️ group_code הרשמי נשמר בטבלת transactions (כי יכול להשתנות בין קבצים)
-- latest_group_code הוא רק cache לנוחות — מתעדכן בכל העלאת קובץ
-- כשמעלים קובץ חדש: UPDATE accounts SET name = X, latest_group_code = Y WHERE code = Z
```

### transactions
```sql
CREATE TABLE transactions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  file_id UUID REFERENCES uploaded_files(id) ON DELETE CASCADE,
  account_id UUID REFERENCES accounts(id),
  group_code TEXT NOT NULL, -- "600","700","702" — הסיווג כפי שהופיע בקובץ המקורי
  original_account_name TEXT, -- שם החשבון כפי שהופיע בקובץ (לתיעוד, כי שם יכול להשתנות בין שנים)
  transaction_date DATE NOT NULL,
  value_date DATE,
  debit NUMERIC(15,2) DEFAULT 0,
  credit NUMERIC(15,2) DEFAULT 0,
  description TEXT,
  counter_account TEXT, -- קוד חשבון נגדי
  reference_number TEXT, -- אסמכתא
  header_number TEXT, -- כותרת
  movement_number TEXT, -- מספר תנועה
  UNIQUE(user_id, header_number, movement_number, account_id)
  -- ↑ מונע כפילויות כשמעלים שנתי + חודשי
);
-- ⚠️ group_code נשמר פה ולא ב-accounts כי ספק/חשבון יכול לעבור סיווג בין שנים
-- למשל: "הוצ חשמל" היה group_code 702 ב-2023 ועבר ל-716 ב-2024
```

### transaction_overrides (שכבת עריכה)
```sql
CREATE TABLE transaction_overrides (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  transaction_id UUID REFERENCES transactions(id) ON DELETE CASCADE,
  user_id UUID REFERENCES auth.users(id),
  override_type TEXT NOT NULL CHECK (override_type IN ('amount', 'category', 'note', 'exclude')),
  original_value TEXT, -- הערך המקורי (לתיעוד)
  new_value TEXT, -- הערך החדש
  note TEXT, -- הערה של המשתמש למה שינה
  created_at TIMESTAMPTZ DEFAULT now()
);
-- override_type = 'exclude' → התנועה לא נספרת בחישובים
-- override_type = 'amount' → new_value = הסכום המתוקן
-- override_type = 'note' → הערה בלבד
```

### custom_groups (קיבוצים מותאמים)
```sql
CREATE TABLE custom_groups (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  name TEXT NOT NULL, -- "עלות ייצור - קניות חומר גלם", "תפעול מפעל"
  display_order INTEGER DEFAULT 0,
  color TEXT, -- צבע לגרפים
  account_codes TEXT[] NOT NULL DEFAULT '{}', -- ["2000","2009","2200","2001",...] ← מפתחות חשבון ספציפיים
  parent_section TEXT NOT NULL CHECK (parent_section IN (
    'revenue', 'cost_of_goods', 'operating', 'admin', 'finance', 'other'
  ))
);
-- ⚠️ שינוי חשוב: הסיווג הוא לפי מפתח חשבון (account code), לא לפי group_code!
-- הסיבה: בדוח רו"ה של רואה החשבון, חשבונות מאותו group_code יכולים להופיע בקבוצות שונות.
-- למשל: 2505 ו-2830 שניהם group_code 703, אבל שניהם תחת "תפעול מפעל" ולא "אחזקה ותפעול"
-- ראה מסמך account-mapping.md לרשימה המלאה של ברירות המחדל
```

### account_classification_overrides (מפתח סיווג פנימי)
```sql
CREATE TABLE account_classification_overrides (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  account_id UUID REFERENCES accounts(id),
  custom_group_id UUID REFERENCES custom_groups(id),
  note TEXT, -- הסבר למה שונה, למשל "אחזקת קו ייצור — שייך לעלות המכר"
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(user_id, account_id) -- רק override אחד לחשבון
);
-- ⚠️ טבלה זו מאפשרת למשתמש להעביר חשבון ספציפי מהקבוצה שלו לקבוצה אחרת
-- למשל: חשבון 2505 "הוצ אחזקה" (group_code 703 = אחזקה ותפעול)
--        המשתמש מחליט שזה בעצם עלות המכר כי זה אחזקת קו ייצור
--        אז הוא יוצר override שמצביע ל-custom_group "עלות המכר"
-- ה-override גובר תמיד על הסיווג מהקובץ, בכל מצב תצוגה (אחרון/מקורי)
```

## לוגיקת הפרסור (Parsing)

### פונקציה: parseAccountingFile(file) → { accounts[], transactions[] }

```
לכל שורה בקובץ:

1. אם עמודה 0 מכילה טקסט ועמודה 1 מכילה קוד מספרי/טקסט ועמודה 2 מכילה group_code:
   → זו שורת כותרת חשבון
   → שמור: currentAccount = { code: col1, name: col0, group_code: col2 }
   → account_type = group_code שמתחיל ב-6 ? 'revenue' : 'expense'
   → בדוק אם החשבון כבר קיים ב-accounts:
     אם קיים → עדכן name ו-latest_group_code (כי הקובץ החדש הוא האמת)
     אם לא קיים → הכנס חדש עם latest_group_code = col2

2. אם עמודה 2 = "יתרת פתיחה":
   → דלג

3. אם עמודה 0 = "סה"כ מפתח חשבון":
   → דלג על שורה זו + 2 שורות הבאות

4. אם עמודה 0 = "סה"כ לדו"ח" או "מספר תנועות בדו"ח":
   → דלג

5. אם עמודות הכותרת (שורה 3 בקובץ): col3="כותרת", col4="תנועה"
   → דלג (זו שורת headers)

6. אם עמודה 0 ועמודה 1 ריקות ויש תאריך בעמודה 8:
   → זו שורת תנועה רגילה
   → חלץ:
     header_number = col3
     movement_number = col4
     counter_account = col7
     transaction_date = col8
     value_date = col9
     reference_number = col10
     description = col12
     debit = col14 (אם ריק → 0)
     credit = col15 (אם ריק → 0)
     group_code = currentAccount.group_code  ← ⚠️ חשוב! נשמר מהקובץ המקורי
     original_account_name = currentAccount.name  ← שם כפי שהופיע בקובץ
   → הוסף ל-transactions עם account_id של currentAccount

7. כל שאר השורות → דלג
```

### עדכון accounts בעת ייבוא

⚠️ **סדר חשוב:** אם המשתמש מעלה קובץ חדש יותר (למשל 2024 אחרי 2023), טבלת accounts מתעדכנת:

```
UPSERT INTO accounts (user_id, code, name, latest_group_code, account_type)
VALUES ($user_id, $code, $name, $group_code, $type)
ON CONFLICT (user_id, code) DO UPDATE SET
  name = EXCLUDED.name,
  latest_group_code = EXCLUDED.latest_group_code;
```

כך `latest_group_code` תמיד משקף את הקובץ האחרון שהועלה.

**אבל:** אם המשתמש מעלה קודם 2024 ואז 2023 — הוא ידרוס את ה-latest עם ערך ישן.
פתרון: בדוק את שנת הקובץ. עדכן latest רק אם שנת הקובץ החדש >= שנת הקובץ האחרון:

```
ON CONFLICT (user_id, code) DO UPDATE SET
  name = CASE WHEN $file_year >= (
    SELECT MAX(year) FROM uploaded_files WHERE user_id = $user_id AND status = 'completed'
  ) THEN EXCLUDED.name ELSE accounts.name END,
  latest_group_code = CASE WHEN $file_year >= (
    SELECT MAX(year) FROM uploaded_files WHERE user_id = $user_id AND status = 'completed'
  ) THEN EXCLUDED.latest_group_code ELSE accounts.latest_group_code END;
```

### מניעת כפילויות

בעת ייבוא, לכל תנועה בדוק:
```sql
SELECT id FROM transactions
WHERE user_id = $user_id
AND header_number = $header_number
AND movement_number = $movement_number
AND account_id = $account_id
```
אם קיים → דלג. אם לא → הכנס.

זה מאפשר להעלות קובץ שנתי ואח"כ חודשי (או להיפך) בלי כפילויות.

## חישובי הדוח

### מצב סיווג (Classification Mode) — 3 שכבות

⚠️ **עקרון מרכזי:** יש 3 שכבות סיווג, בסדר עדיפות:

```
שכבה 1 (גובר על הכל): account_classification_overrides
  → המשתמש העביר חשבון ספציפי לקבוצה אחרת ידנית
  → אם קיים override — הוא תמיד גובר, בכל מצב תצוגה
  → דוגמה: "הוצ אחזקה" (703) → המשתמש העביר ל-"עלות המכר"

שכבה 2 (ברירת מחדל):  custom_groups לפי effective_group_code
  → מצב "אחרון": effective_group_code = accounts.latest_group_code
  → מצב "מקורי": effective_group_code = transactions.group_code
  → דוגמה: "הוצ חשמל" group_code 702 → נכנס ל-custom_group "הוצאות ייצור"

שכבה 3 (הבסיס):       group_code מהקובץ המקורי
  → נשמר בטבלת transactions, לא ניתן לשינוי
  → זה ה-fallback אם אין custom_groups מוגדרים
```

**הפונקציה המרכזית:**
```
getAccountCategory(account, transaction, mode):

  // שכבה 1: בדוק override פנימי — גובר תמיד
  override = SELECT custom_group_id
             FROM account_classification_overrides
             WHERE account_id = account.id AND user_id = $user_id
  if override exists:
    → return override.custom_group

  // שכבה 2: לפי account code ב-custom_groups
  custom_group = SELECT * FROM custom_groups
                 WHERE account.code = ANY(account_codes)
                 AND user_id = $user_id
  if found:
    → return custom_group

  // שכבה 3: fallback — חשבון שלא שויך לאף קבוצה
  → return "אחר" (other)
```

**ב-UI:** toggle קטן בפינת הדשבורד/טבלה:
```
סיווג: [● אחרון | ○ מקורי]
```
ברירת מחדל = "אחרון".

⚠️ **שים לב:** כשיש override פנימי (שכבה 1), ה-toggle לא משפיע על אותו חשבון — הוא תמיד יופיע בקבוצה שהמשתמש בחר. ה-toggle משפיע רק על חשבונות ללא override.

### הכנסות
```
הכנסות = סה"כ credit בחשבונות עם effective_group_code שמתחיל ב-"6"
         (כולל 600, 601, וכו')
         פחות: סה"כ debit בחשבון 699 (החזרות)
```

  "תפעול מפעל"                      → account_codes: ["2830","2538","2201","2206","2505","2506"]

⚠️ **חשבונות שלא שויכו:** כשמעלים כרטסת ונמצא חשבון שלא מופיע באף custom_group → הוא נכנס ל-"אחר" ומופיעה התראה למשתמש לסווג אותו.

### חישוב לכל חודש
```
לכל חודש (1-12):
  סנן transactions לפי EXTRACT(MONTH FROM transaction_date) = month
  לכל transaction:
    category = getAccountCategory(account, transaction, mode)
    // ↑ בודק override פנימי (שכבה 1) → custom_groups (שכבה 2) → group_code (שכבה 3)
    amount = debit - credit
    (בחשבונות הכנסה: amount = credit - debit)
  קבץ לפי category (= custom_group שחזר מהפונקציה)
```

⚠️ **שים לב:** חשבון עם override פנימי תמיד ייכנס לקבוצה שהמשתמש בחר, גם אם ה-toggle במצב "מקורי".

### טיפול ב-overrides
```
לפני חישוב סכום תנועה, בדוק:
  - אם יש override מסוג 'exclude' → דלג על התנועה
  - אם יש override מסוג 'amount' → השתמש ב-new_value במקום הסכום המקורי
```

לוגיקת זיהוי סטיות:
```
לכל חשבון, לכל חודש:
  ממוצע = AVG(amount) של כל החודשים
  סטיית_תקן = STDDEV(amount)

  אם amount > ממוצע + 1.5 × סטיית_תקן → צהוב
  אם amount > ממוצע + 2.0 × סטיית_תקן → אדום

לכל חשבון, השוואה שנתית (אם יש 2 שנים):
  שינוי = (סה"כ_שנה_נוכחית - סה"כ_שנה_קודמת) / סה"כ_שנה_קודמת × 100
  אם שינוי > 30% → צהוב
  אם שינוי > 50% → אדום

עליות רצופות:
  אם 3+ חודשים רצופים שכל אחד גבוה מהקודם → צהוב
```

## עיצוב ו-UX

### סגנון כללי
- **מודרני, נקי, מינימליסטי** — הרבה רווח לבן
- **כיוון RTL** — עברית, טקסט מימין לשמאל
- **צבעים:**
  - ירוק (#10B981) = רווח / שיפור / חיסכון
  - אדום (#EF4444) = הפסד / הידרדרות / חריגה
  - צהוב (#F59E0B) = שים לב
  - כחול (#3B82F6) = מידע / ניטרלי
  - אפור (#6B7280) = טקסט משני
- **פונט:** בלי סריפים, תומך עברית (Heebo או Assistant מ-Google Fonts)
- **מספרים:** תמיד עם פסיקים (1,234,567) ותמיד ₪ לפני
- **אחוזים:** תמיד עם סימן כיוון (▲+12%, ▼-5%, ═0%)

### אינטראקציות
- **Hover על מספר** → tooltip עם פירוט
- **לחיצה על מספר** → drill down
- **Side panels** נפתחים עם אנימציית slide מימין
- **Modals** לפעולות עריכה
- **Drag & drop** לתגיות וקיבוצים
- **Skeleton loading** בזמן טעינה (לא spinner)
- **Toast notifications** לאישור פעולות

### רספונסיביות
- Desktop: layout מלא עם side panels
- Tablet: side panel מתחת במקום בצד
- Mobile: כרטיסים אחד מתחת לשני, טבלה עם scroll אופקי
